Symbolication is the process of translating minified production stack traces back into the original source locations your team actually wrote. Browsers report errors against the shipped bundle, so without symbolication an exception looks like this:

```
TypeError: Cannot read properties of undefined (reading 'name')
anonymous()
    app.bundle.js:1:48211
anonymous()
    app.bundle.js:1:13377
```

With source maps uploaded, Traceway stores it like this instead:

```
TypeError: Cannot read properties of undefined (reading 'name')
renderUserCard()
    src/components/UserCard.tsx:42:17
handleClick()
    src/pages/Profile.tsx:108:5
```

This page explains exactly how that translation works today, what is cached and when, how the pipeline is observable through metrics, where the current limits are, and how we verify the output. For how to upload source maps in the first place, see [Source Maps](/client/js-sdk/sourcemap-upload).

## When Symbolication Runs

Symbolication happens **once, at ingest**, inside `POST /api/report`:

1. The report arrives from the JS SDK. If the project is a JS framework, each frame is resolved directly against object storage at `sourcemaps/{projectId}/{filename}`: the map is addressed by the frame's filename, with no version and no database lookup. The most recent upload of each filename wins.
2. Each stack trace in the report is resolved frame by frame.
3. The **resolved** trace is what gets hashed for grouping and stored in ClickHouse.

The same resolution runs for exceptions arriving over the OTLP endpoint when the resource reports a JavaScript SDK (`telemetry.sdk.language` of `webjs`, `nodejs`, `javascript`, or `typescript`).

Nothing is re-resolved when you view an issue. The trade-off: traces are immutable snapshots of the symbolication quality at the moment they arrived. Uploading better source maps improves future events, not past ones.

## The Frame Format

The JS SDK does not send the browser's raw `error.stack` string. Stack formats are not standardized: V8 (Chrome, Edge, Hermes) emits `    at funcName (file:line:col)`, while Firefox and JavaScriptCore (Safari) emit `funcName@file:line:col`. The SDK parses both families and normalizes every frame into a canonical two-line wire format: a function name line ending in `()`, then a location line indented by four spaces:

```
anonymous()
    app.bundle.js:1:48211
```

The resolver recognizes location lines with a single regex:

```
^(\s{4})(.+):(\d+):(\d+)$
```

Anything that does not match (the error message, the name lines, blank lines) passes through untouched. This is the first invariant of the pipeline: **the resolver can only rewrite a line or leave it exactly as it was**. There is no failure mode that drops or corrupts a frame.

## The Resolution Pipeline

For each location line, in order:

### 1. Find the right source map

The frame's file name is reduced to its basename (any query string or fragment stripped first), and the map is addressed in object storage at a deterministic key:

| Frame file | Storage key |
|---|---|
| `app.bundle.js` | `sourcemaps/{projectId}/app.bundle.js.map` |
| `https://cdn.example.com/assets/app.bundle.js` | `sourcemaps/{projectId}/app.bundle.js.map` |
| `app.bundle.js?v=abc123` | `sourcemaps/{projectId}/app.bundle.js.map` |

The minified bundle, when the upload CLI found one next to the map, lives at the same key without the `.map` suffix and feeds function name resolution (next section).

No map in storage: the line passes through unchanged, and that filename enters a miss cooldown so subsequent exceptions skip the storage read entirely (see "When loading fails" below).

### 2. Get the resolver

The parsed lookup structure comes from the in-memory cache described below. First use pays a storage read plus a parse; everything after is a memory lookup.

### 3. Look up the position

Browser stack traces use 1-based columns; the source map spec uses 0-based. The resolver subtracts 1 before the lookup and adds 1 back when printing. This off-by-one matters more than it sounds: minified bundles put the entire program on line 1, so being one column off can land on a *different token entirely* and resolve to the wrong original file.

Here is a real lookup, taken verbatim from our test fixtures. The original file:

```js
// built with: terser -c -m --module original.js --source-map includeSources
function abcd() {}
export default abcd;
```

terser renames `abcd` to `t` and emits:

```js
function t(){}export default t;
```

with this source map:

```json
{
  "version": 3,
  "names": ["abcd"],
  "sources": ["tests/fixtures/simple/original.js"],
  "sourcesContent": ["...the original file, embedded..."],
  "mappings": "AACA,SAASA,oBACMA"
}
```

A frame pointing at `minified.js:1:11` (the `t` in `function t`) resolves through the `mappings` (delta-encoded VLQ segments of generated position, source index, original position, name index) to `tests/fixtures/simple/original.js:2:10`, with the name token `abcd`.

### 4. Rewrite the lines

The location line becomes `original-file:line:column` (same indentation). If a function name was resolved, the *name line directly above it* (the `anonymous()` line) is rewritten to `abcd()`.

If the source map carries `"sources": [null]` entries (older uglify pipelines), the file name is rendered as `<unknown>` but the line, column, and name still resolve.

A maximum of **50 frames** per trace are resolved; deeper frames pass through minified.

## Function Names: Scope Analysis

Locations come straight from the source map and are exact. Function names are harder, because a source map's `names` token at a position is the identifier *at that token*, not the name of the *enclosing function* that a stack trace conventionally shows. Getting the enclosing function right requires scope analysis: parsing the minified bundle into an AST and mapping positions to function scopes. This is the same approach Sentry's symbolic takes, and it is why the upload CLI sends the minified bundle alongside each `.map`.

When a bundle is available, the resolver:

1. Parses it with a pure-Go JavaScript parser into every function in the file: declarations, function expressions, arrow functions, and class methods, each with the generated position of its name. If the bundle fails to parse as a script (ESM syntax), it is retried with module syntax neutralized.
2. Flattens the nested scopes into a sorted list of transitions: "from this generated position onward, the innermost enclosing function's name sits at line L, column C".
3. For each mapping in the source map, resolves the enclosing function's *name position* back through the map itself, yielding the original, pre-minification name.

Take preact's `create-element.js`, which the old name heuristic famously got wrong:

```js
export function createElement(type, props, children) {
  let normalizedProps = {},
    key, ref, i;
  for (i in props) {                         // error around here (line 40)
    ...
  }
}
```

A frame inside that loop now resolves to `createElement`, because the scope walk knows which function *contains* the error position rather than guessing from nearby text.

When no bundle was uploaded (a map-only upload from an older CLI, or a `.map` with no sibling bundle in the build output), there is no scope information and **no name is resolved**: the name line stays whatever the SDK reported (typically `anonymous()`), while file, line, and column still resolve exactly. Two design decisions contain the remaining gaps:

1. **Names never affect grouping.** See the next section.
2. **Locations never depend on the bundle.** The file, line, and column shown are exact source map output, so the click-through is right even when the label is missing.

## How Symbolication Interacts with Grouping

Issue grouping hashes a normalized form of the **resolved** trace. Two normalization rules exist specifically because of JS symbolication:

**Function name lines are collapsed.** A line ending in `()` that sits directly above a frame location line becomes `<fn>` before hashing (anchoring on the location line below means Go traces, whose file lines are tab-indented, are never touched):

```
createElement()                       becomes:    <fn>
    ../src/create-element.js:40:9     stays:      ../src/create-element.js:40:9
```

So these two traces, identical except for resolved names, produce the **same hash** and land in the same issue:

```
Error: boom                          Error: boom
    normalizedProps()                    createElement()
    ../src/create-element.js:40:9        ../src/create-element.js:40:9
```

This means name quality can keep improving (for example, by uploading bundles where you previously uploaded only maps) without re-bucketing your existing issues.

**Columns are kept.** Minified bundles put everything on line 1, so when no source map matched, the column is the only thing distinguishing two different errors. `bundle.min.js:1:48211` and `bundle.min.js:1:91567` therefore hash differently, on purpose.

One consequence to be aware of: a trace that fails symbolication (storage outage, see below) hashes differently than its symbolicated twin. During an outage the same logical error can transiently open a second issue group, which stops growing once storage recovers.

## What Is Cached, How, and When

Source maps for real applications are large (a 20 MB `.map` is unremarkable), and the compiled lookup structure is what actually serves frames. Traceway keeps **one in-memory cache** of built resolvers, keyed by the map's storage key.

### What a cache entry contains

Building a resolver consumes the `.map` and the bundle and keeps only what lookups need:

| Field | Used for |
|---|---|
| token table (one 24-byte token per mapping) | position lookup via binary search |
| interned file names | original file names |
| interned function names | enclosing function names, pre-resolved per scope at build time |

Everything else is discarded: the raw map and bundle bytes, `sourcesContent`, the bundle's AST, and the scope transitions all exist **only for the duration of the build**. Scope analysis runs once per build, not per lookup, so serving a frame is a binary search over the token table. The entry's in-memory size is estimated from the token count plus the interned strings, which is what the byte cap counts.

### Cache policy

| Property | Value |
|---|---|
| Keying | map storage key, one entry per bundle filename per project |
| Capacity | `SOURCEMAP_CACHE_MAX_ENTRIES` (default 200) and `SOURCEMAP_CACHE_MAX_BYTES_MB` (default 500, estimated resolver size) |
| Eviction | least recently used, whenever either cap is exceeded |
| Loading | deduplicated (singleflight): concurrent requests for the same map share one storage read and one build |
| Load timeout | 5 seconds per storage read |
| Failures | negative-cached with an escalating cooldown (below) |
| Invalidation | uploading a file evicts its resolver and clears its cooldown immediately |

### The lifecycle, end to end

Say two exceptions referencing `app.bundle.js.map` arrive at the same time, on a cold cache:

1. Both miss the cache and race to a mutex. The winner registers itself as the **loader** for that key; the loser sees the in-flight registration and parks.
2. The loader reads the `.map` and the bundle from storage (S3 or local disk, 5 second budget per read), builds the resolver (parse map, parse bundle, pre-resolve scope names), throws the raw bytes away, and inserts the slim entry.
3. The parked request wakes and receives the exact same resolver instance. One read, one build, one copy in memory, no matter how many requests piled up.
4. Every subsequent exception is a pure memory hit, which also bumps the entry to the front of the LRU.
5. If the cache later exceeds a cap, the least recently used entries are dropped. A dropped resolver is not an error; the next exception that needs it repeats steps 1 to 3.
6. Re-uploading `app.bundle.js.map` (or `app.bundle.js`) evicts the cached resolver, so the next exception rebuilds from the new upload rather than serving the stale one.

### When loading fails

If the map does not exist, the read times out, storage errors, or the map fails to parse:

- The affected frames pass through **unchanged**, and the trace is stored as-is. Ingest is never blocked beyond the per-read budgets (5 seconds each for the map and the bundle) and never errors because of symbolication. A transient failure reading the *bundle* fails the build too, rather than silently caching a resolver without function names; the cooldown retries it within seconds.
- The failure is **negative-cached per filename**: while the cooldown is active, frames referencing that filename skip the storage read entirely. A missing map starts at a 1 minute cooldown; a transient failure (timeout, storage error, unparseable map) starts at 15 seconds, so brief storage hiccups recover fast. Each consecutive failure doubles the cooldown, capped at 15 minutes, so a project that never uploads maps converges to a few storage reads per hour per filename instead of one per minute.
- The cooldown clears the moment that filename is uploaded (the upload endpoint invalidates it in-process) or a load succeeds. In a multi-instance deployment, instances other than the one that served the upload retry within their remaining cooldown, at most 15 minutes.
- The negative cache is bounded at 10,000 filenames; expired entries are pruned when the bound is hit.
- Within a single trace, a failed map is attempted **once**, not once per frame, so a 50-frame trace cannot stack fifty timeouts.
- Traceway self-reports the failure into its own monitoring project, throttled to one exception per minute with a count of suppressed occurrences.

## Observability

The backend self-reports cache metrics every 30 seconds (see [Capacity Monitoring](/server/capacity-monitoring) for the full operations view):

| Metric | Meaning |
|---|---|
| `traceway.sourcemap.entries` | built resolvers currently in memory |
| `traceway.sourcemap.bytes` | estimated total size of those entries |
| `traceway.sourcemap.negative_entries` | filenames currently under a miss/failure cooldown |
| `traceway.sourcemap.hits.delta` | cache hits since the last tick |
| `traceway.sourcemap.misses.delta` | builds triggered since the last tick |
| `traceway.sourcemap.evictions.delta` | LRU evictions since the last tick |
| `traceway.sourcemap.load_failures.delta` | failed builds (storage error, 5s timeout, unparseable map); each one is a trace stored unsymbolicated |
| `traceway.sourcemap.not_found.delta` | lookups that found no uploaded map for the frame's filename since the last tick |
| `traceway.sourcemap.negative_hits.delta` | frame lookups skipped by an active cooldown since the last tick |
| `traceway.sourcemap.parse_ms` | duration of the most recent resolver build |

What healthy looks like: `misses.delta` near zero in steady state (every miss is a storage read plus a rebuild), `load_failures.delta` at zero (anything above zero means minified traces are reaching your issues), and `bytes` comfortably under the cap. A useful alert is `load_failures.delta` max greater than 0 over 5 minutes.

`not_found.delta` is a configuration signal rather than a failure signal: sustained nonzero values for a project that should be symbolicated mean exceptions reference filenames that were never uploaded, most commonly a missing upload step in CI or a `--directory` pointing at the wrong build output. `negative_hits.delta` shows the cooldown doing its job; high values alongside high `not_found.delta` are normal for JS projects that simply do not upload maps.

## Limitations Today

Honest list, in rough order of impact:

1. **Function names require the bundle.** The current upload CLI sends it automatically, but map-only uploads (older CLI versions, or maps whose sibling bundle was not in the build output) resolve locations without names.
2. **Symbolication is ingest-only.** Maps uploaded after an event arrived do not improve that event.
3. **Maps are matched by basename.** Two bundles with the same filename in different directories collide, and the most recent upload wins. Content-hashed bundle names (the default in Vite, Next, Angular, and most bundlers) avoid this entirely; stable names like `app.js` shared across concurrently deployed versions resolve against the latest upload only.
4. **The 5 second load budget trades completeness for latency.** A map that takes longer to fetch produces an unsymbolicated trace (recorded in `load_failures.delta`) rather than a slow ingest.
5. **Failure cooldowns can delay recovery on other instances.** Uploading a map clears its cooldown instantly on the instance that handled the upload; in multi-instance deployments, the rest retry within their remaining cooldown, at most 15 minutes.
6. **Outages split grouping transiently.** Unsymbolicated and symbolicated versions of the same error hash differently.
7. **Maps with `null` sources are ambiguous.** Frames resolve as `<unknown>`, though line, column, and name still resolve.
8. **50 resolved frames per trace.** Deeper frames stay minified.
9. **Module-level code has no name.** Errors thrown outside any function (Hermes/Metro global prologue, top-level module code) resolve location-only rather than being labeled `<global>`.

## How We Verify It

The resolver is tested for parity against [getsentry/symbolic](https://github.com/getsentry/symbolic), the symbolication engine behind Sentry, using its own fixture suite (webpack, Metro, Hermes, preact, inlined functions, null-source maps). Every fixture frame's expected file, line, column, and name is asserted exactly, so any drift fails the build.

All frames match symbolic on **location**. With scope analysis, names now match too for frames inside a function, including the cases the old text-scanning heuristic got wrong (preact's `createElement` and `assign` among them). The remaining name gap is module-level code, where symbolic reports `<global>` and Traceway reports no name (limitation 9 above).
