# Alerts

Alerts notify you when conditions in your application require attention — errors, latency spikes, metric thresholds, or missing data. Configure channels and rules to get notified through email, Slack, webhooks, or GitHub Issues.

## How Alerts Work

The alerting system has two parts:

- **Channels** define where notifications are delivered (email, Slack, webhook, GitHub)
- **Rules** define what conditions trigger a notification

When a rule's condition is met, Traceway sends a notification through the rule's attached channel. Each rule is linked to exactly one channel, but a channel can be used by multiple rules.

## Channels

A channel represents a notification destination. Traceway supports four channel types:

| Channel | Description | Configuration |
|---------|-------------|---------------|
| Email   | Sends an email to one or more recipients | List of email addresses |
| Slack   | Posts a message to a Slack channel via incoming webhook | Webhook URL |
| Webhook | Sends an HTTP POST with a JSON payload to any URL | URL, optional headers, optional HMAC secret |
| GitHub  | Creates a GitHub issue in a repository | Personal access token, repository owner/name, optional labels |

Each channel has a **Test** button that sends a sample notification so you can verify the configuration before attaching rules.

### Webhook Payload

When a webhook channel fires, Traceway sends a JSON POST request with the following fields:

| Field | Description |
|-------|-------------|
| `subject` | Alert title |
| `body` | Alert details |
| `severity` | `info`, `warning`, or `critical` |
| `ruleType` | The rule type that triggered the alert |
| `ruleName` | The name you gave the rule |
| `url` | Deep link to the relevant dashboard page |
| `timestamp` | When the alert fired (ISO 8601) |

If you configure an HMAC secret on the channel, each request includes an `X-Traceway-Signature` header containing `sha256=<hex>` of the request body, which you can use to verify that the request came from Traceway.

## Rules

### Rule Types

| Rule | What It Monitors |
|------|-----------------|
| New Issue | A never-before-seen exception appears |
| Impact Score Critical | An endpoint's [impact score](/learn/endpoints#impact-score) reaches critical level |
| Impact Score High | An endpoint's impact score reaches high level |
| Impact Score Medium | An endpoint's impact score reaches medium level |

### Coming Soon

The following rule types are planned but not yet available:

- **Error Regression** — A previously resolved (archived) [issue](/learn/issues) reappears
- **Error Rate** — Overall 5xx error rate exceeds a percentage threshold
- **Error Count** — Total number of errors exceeds a count in a time window
- **Endpoint P95** — P95 latency for an endpoint exceeds a millisecond threshold
- **Endpoint P99** — P99 latency for an endpoint exceeds a millisecond threshold
- **Endpoint Error Rate** — A specific endpoint's error rate exceeds a percentage threshold
- **Apdex Drop** — Apdex score falls below a threshold
- **Metric Threshold** — A custom or system [metric](/learn/metrics) crosses a boundary (above or below)
- **No Data** — No telemetry data received for a specified duration
- **Task Duration** — A background task's P95 duration exceeds a threshold
- **Task Failure Rate** — A background task's failure rate exceeds a percentage threshold
- **Throughput Drop** — Request volume drops compared to baseline

### Evaluation

New Issue rules are evaluated immediately when new data arrives during report ingestion. Impact Score rules are evaluated every 60 seconds by a background process.

## Cooldown and Snooze

**Cooldown** prevents a rule from firing repeatedly for the same ongoing condition. After an alert fires, the rule will not fire again until the cooldown period expires. The default cooldown is 15 minutes, and it can be configured per rule.

**Snooze** lets you temporarily mute a rule for a set duration from the dashboard. This is useful during deployments, maintenance windows, or known incidents where you don't want to be notified.

## Severity

Each alert is assigned a severity level based on the condition:

| Severity | Typical Triggers |
|----------|-----------------|
| Critical | New errors, critical impact score |
| Warning  | High impact score |
| Info     | Medium impact score |
