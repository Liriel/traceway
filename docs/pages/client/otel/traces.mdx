# Traces

Traceway ingests OpenTelemetry spans via the OTLP/HTTP protocol and maps them to its own data model.

## Span Mapping

Spans are classified by **kind and attributes**, not by whether they're a root span. Both root and child spans can be promoted to a top-level concept when the signal is there — a queued job processed as a child of the dispatcher's trace still becomes a Task, an LLM call made inside a request handler still becomes an AI Trace.

| Condition | Traceway Concept |
|-----------|-----------------|
| `SpanKind = SERVER` / `INTERNAL` with HTTP attributes, and the span is a root (or its parent is in another process — a cross-service inbound hop) | **Endpoint** |
| `SpanKind = CONSUMER` (queue / messenger workers) | **Task** |
| Root `INTERNAL` span with a `console.command` attribute (artisan commands, console runners) | **Task** |
| Any span with `gen_ai.*` attributes (root or child of a request/task) | **AI Trace** |
| Any other span with a parent | **Span** (waterfall row under its nearest enclosing entity) |
| Event named `"exception"` on any span | **Issue** (attributed to the owning entity) |

Each Endpoint / Task / AI Trace row carries an `is_root` flag. Non-root rows surface a **Non-root** chip in the dashboard list so you can tell at a glance which runs were triggered by another trace, and a **View distributed trace** link on the detail page jumps you to the full waterfall (endpoint → producer → consumer, request → child LLM call, etc.). The `distributed_trace_id` linking the rows is auto-derived from the OTel `trace_id`, so cross-trace lookup works without any vendor attribute.

> **Sub-handler spans are not duplicated.** An in-process child SERVER/INTERNAL+HTTP span (e.g. a framework's "handler /path" sub-span) is kept in the Spans waterfall — Endpoints are only promoted for entry points (root or cross-service hop).

## Resource Attributes

These resource-level attributes are extracted and applied to all spans in the resource:

| Attribute | Traceway Field | Description |
|-----------|---------------|-------------|
| `service.name` | Server Name | Identifies the service or application |
| `service.version` | App Version | Application version string |

## HTTP Semantic Conventions

For root SERVER spans with HTTP attributes, Traceway extracts the following:

| OTel Attribute | Fallback | Traceway Field |
|---------------|----------|----------------|
| `http.request.method` | `http.method` | HTTP method |
| `http.route` | Span name | Route path |
| `http.response.status_code` | `http.status_code` | Status code |
| `http.response.body.size` | `http.response_content_length` | Response body size |
| `client.address` | `net.peer.ip` | Client IP |

The endpoint name is constructed as `"METHOD /route"` (e.g., `"GET /api/users"`).

### Streaming Responses (SSE / WebSocket)

Long-lived streaming routes are flagged as `is_stream` on ingest so that the dashboard excludes them from P50/P95/P99, Apdex, and impact scoring. They still appear in the endpoints list with their count, status code, error rate, and throughput. Detection signals (any of):

| Signal | OTel attribute |
|--------|----------------|
| Server-Sent Events | `http.response.header.content-type` contains `text/event-stream` |
| WebSocket upgrade | `http.response.status_code = 101` |
| Vendor opt-in | `traceway.is_stream = true` (bool) — for protocols the headers don't expose |

There is no OpenTelemetry-standard streaming attribute; Traceway sniffs the captured Content-Type response header. Make sure your OTel instrumentation actually captures `http.response.header.content-type` (some SDKs require explicit opt-in for header capture). Otherwise set `traceway.is_stream = true` directly on the span when starting a stream.

## Exception Events

Spans with events named `"exception"` are extracted as Issues in Traceway. The following event attributes are used:

| Attribute | Description |
|-----------|-------------|
| `exception.type` | Error class name (e.g., `RuntimeError`) |
| `exception.message` | Error message |
| `exception.stacktrace` | Full stack trace |

Exceptions are grouped by a normalized hash of the stack trace, so the same logical error produces the same Issue regardless of runtime differences like memory addresses or user IDs.

## Example: Node.js with OTel SDK

```typescript
import { NodeSDK } from "@opentelemetry/sdk-node";
import { OTLPTraceExporter } from "@opentelemetry/exporter-trace-otlp-http";
import { getNodeAutoInstrumentations } from "@opentelemetry/auto-instrumentations-node";
import { Resource } from "@opentelemetry/resources";

const sdk = new NodeSDK({
  resource: new Resource({
    "service.name": "my-service",
    "service.version": "1.0.0",
  }),
  traceExporter: new OTLPTraceExporter({
    url: "https://your-traceway-instance.com/api/otel/v1/traces",
    headers: {
      Authorization: "Bearer your-project-token",
    },
  }),
  instrumentations: [getNodeAutoInstrumentations()],
});

sdk.start();
```

This automatically instruments HTTP servers, database clients, and other libraries. SERVER spans become Endpoints, CONSUMER spans become Tasks, any span with `gen_ai.*` attributes becomes an AI Trace (root or child), and exception events become Issues.

## Example: Python with OTel SDK

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource

resource = Resource.create({
    "service.name": "my-python-service",
    "service.version": "1.0.0",
})

provider = TracerProvider(resource=resource)
exporter = OTLPSpanExporter(
    endpoint="https://your-traceway-instance.com/api/otel/v1/traces",
    headers={"Authorization": "Bearer your-project-token"},
)
provider.add_span_processor(BatchSpanProcessor(exporter))
trace.set_tracer_provider(provider)
```
